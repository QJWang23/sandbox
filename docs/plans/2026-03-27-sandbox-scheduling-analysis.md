# 沙箱高并发调度：无状态轻量级调度与 K8s 传统调度的核心技术差异分析

## 1. 问题定义与背景

### 1.1 沙箱高并发场景的核心挑战

智能体（Agent）沙箱的调度模式与 K8s 传统工作负载存在本质差异：

| 特征 | 传统 K8s 工作负载 | 智能体沙箱 |
|------|-----------------|-----------|
| 生命周期 | 分钟~月 | 秒~分钟 |
| 创建频率 | 偶发（部署/扩缩容） | 持续高频（每次 Agent 交互） |
| 创建延迟容忍 | 秒级可接受 | 要求毫秒级（用户体验直接相关） |
| 并发规模 | 数十~数百 Pod | 单用户可触发 10+ 沙箱，集群级万级并发 |
| 资源模型 | 静态 Request/Limit | 动态波动（推理计算密集 vs 等待状态） |
| 状态 | 大量有状态服务 | 无状态执行单元（可快速创建销毁） |

**核心矛盾：K8s 调度器为长期稳定运行的工作负载设计，其每一步决策都经过重的状态共识路径，这在沙箱高频创建销毁场景下成为性能瓶颈。**

### 1.2 现有方案的性能基准

基于项目前期调研数据：

| 方案 | 冷启动延迟 | 预热池延迟 | 并发创建能力 |
|------|-----------|-----------|------------|
| **E2B** | 150ms | ~50ms | 150 个/秒/主机 |
| **OpenKruise/agents** | ~6s | 300-600ms | 10-20 个/秒/节点 |
| **k8s agent-sandbox** | ~6s | 300-600ms | 10-20 个/秒/节点 |

E2B 的 150ms 启动远优于 K8s 原生方案，其根本原因不在于运行时技术差异（两者都可用 Firecracker），而在于**调度路径的架构差异**。

## 2. 核心技术差异分析

### 2.1 调度路径对比

#### K8s 传统调度路径

```
沙箱创建请求
    │
    ▼
┌─────────────────┐     ~5-10ms
│  API Server      │────▶ etcd 写入（状态持久化）
│  (请求校验/鉴权)  │────▶ Admission Webhook（可选）
└────────┬────────┘
         │ Pod 对象入队
         ▼
┌─────────────────┐     ~50-200ms
│  kube-scheduler  │────▶ 遍历所有候选节点（Filter）
│  (调度框架)      │────▶ 评分排序（Score）
└────────┬────────┘────▶ 选择最优节点（Bind）
         │
         ▼
┌─────────────────┐     ~10-20ms
│  API Server      │────▶ 更新 Pod .spec.nodeName
└────────┬────────┘────▶ etcd 写入
         │
         ▼
┌─────────────────┐     ~500ms-3s
│  kubelet         │────▶ CRI 调用容器运行时
│  (目标节点)      │────▶ 拉取镜像（如未缓存）
└────────┬────────┘────▶ 创建容器/VM
         │
         ▼
    沙箱就绪

总计延迟: 600ms ~ 5s+
```

**延迟构成分析**：

| 阶段 | 延迟占比 | 是否可优化 | 优化瓶颈 |
|------|---------|-----------|---------|
| API Server + etcd | 15-25% | 有限 | 分布式一致性要求 |
| kube-scheduler | 20-40% | 中等 | 遍历所有节点的 Filter/Score |
| 镜像拉取 | 30-50% | 高 | 网络 I/O，可通过预热池绕过 |
| 容器/VM 创建 | 10-20% | 高 | 运行时初始化，Firecracker 已优化到 125ms |

#### 无状态轻量级调度路径

```
沙箱创建请求
    │
    ▼
┌─────────────────┐     <1ms
│  轻量级调度器     │────▶ 内存级槽位查询（O(1) / O(log n)）
│  (Fast Path)     │────▶ 无 API Server 调用
└────────┬────────┘────▶ 无 etcd 写入
         │
         ▼
┌─────────────────┐     ~5-50ms
│  目标节点        │────▶ 从预热池直接分配
│  (本地激活)      │────▶ 注入用户代码/配置
└────────┬────────┘────▶ 跳过镜像拉取
         │
         ▼
    沙箱就绪

总计延迟: ~5-50ms（热池命中时）
```

### 2.2 六大核心技术差异

#### 差异 1: 状态管理模型

```
K8s 传统调度:
  ┌──────────┐    写入    ┌──────┐    读取    ┌──────────┐
  │ Scheduler │──────────▶│ etcd │◀─────────│ 调度对象   │
  └──────────┘           └──────┘           └──────────┘
       │                   Raft 共识              │
       │                   ~5-10ms               │
  所有状态变更必须经过 etcd → Raft 一致性协议
  → 保证强一致性，但引入固定延迟

无状态轻量级调度:
  ┌──────────┐    内存查询   ┌──────────┐
  │ Scheduler │────────────▶│ 槽位索引  │
  └──────────┘              └──────────┘
  调度器本地维护状态快照（通过心跳异步更新）
  → 最终一致性，但调度延迟为零
```

**决策影响**：沙箱是短暂的执行单元，不需要强一致性保证。一个沙箱分配到"次优"节点的代价远小于每次调度都付出 etcd 延迟的代价。

#### 差异 2: 调度决策复杂度

```
K8s 调度决策（每次 Pod 创建）:
  Filter 阶段:
    ├─ PodFitsResources      # CPU/Memory/GPU 请求量检查
    ├─ PodFitsHostPorts       # 端口冲突检查
    ├─ MatchNodeSelector      # 节点标签匹配
    ├─ NoVolumeZoneConflict   # 存储拓扑检查
    ├─ PodToleratesNodeTaints # 污点容忍检查
    ├─ MaxEBSVolumeCount      # 存储卷限制
    └─ ... 20+ 内置插件       # 还有自定义插件

  Score 阶段:
    ├─ NodeResourcesFit       # 资源碎片化评分
    ├─ NodeAffinity           # 亲和性评分
    ├─ TaintToleration        # 污点容忍评分
    └─ ... 10+ 内置插件

  复杂度: O(N * K)，N = 节点数，K = 插件数
  典型值: N=100, K=30 → 每次调度 3000+ 次检查

轻量级调度决策:
  1. 根据资源标签 → 候选池  O(1)
  2. 候选池取最优节点        O(log N)
  3. 槽位扣减              O(1)

  复杂度: O(log N)
  典型值: N=100 → 每次调度 ~7 次操作
```

**决策影响**：沙箱场景的调度约束远少于传统工作负载。大多数沙箱只关心"有没有空闲槽位"和"什么类型的资源"，不需要 20+ 个过滤插件。

#### 差异 3: 资源模型

```
K8s 资源模型（静态预留）:
  Pod spec:
    resources:
      requests:
        cpu: "2"          # 无论是否使用，都预留
        memory: "4Gi"     # 即使空闲，也不可分配给其他 Pod
        nvidia.com/gpu: "1"

  问题: 推理 Agent 的工作模式是"计算密集 + 等待交替"
  → 计算时需要 GPU → 等待时 GPU 闲置但不可回收
  → 导致资源利用率 20-40%

轻量级调度资源模型（动态感知）:
  槽位 = 预热好的沙箱实例（实际可用资源）

  不预留，只分配"已就绪"的沙箱
  沙箱使用完毕 → 回收 → 重新预热 → 回到池中
  → 资源利用率取决于预热池的水位管理策略
```

**决策影响**：静态预留模型在间歇性工作负载下资源浪费严重。槽位模型将"资源分配"转化为"实例分配"，调度器只需知道"池中有多少可用实例"。

#### 差异 4: 通信路径

```
K8s 路径（请求-响应-通知 三次通信）:
  Client → API Server → (etcd) → Scheduler → (etcd) → API Server → kubelet
    │                                                                          │
    └──────────────────── 至少 4 次网络往返 ────────────────────────────────────┘

轻量级路径（直接通信）:
  Client → Scheduler → 目标节点
    │                        │
    └──── 2 次网络往返 ──────┘
```

#### 差异 5: 扩展瓶颈

```
K8s: API Server 是中心化瓶颈
  - 所有 Pod 创建请求经过同一个 API Server
  - etcd 写入是串行的（Raft 协议）
  - 水平扩展 API Server 可以缓解读压力，但写入仍是瓶颈
  - 高并发场景下 API Server 成为限流点

轻量级: 调度器可水平扩展
  - 调度器无状态，可以多实例部署
  - 每个实例维护本地槽位索引
  - 通过一致性协议（如 Raft for 元数据，非调度决策）保证一致性
  - 理论上吞吐量随实例数线性增长
```

#### 差异 6: 故障域

```
K8s: 调度器是控制平面的一部分
  - kube-scheduler 宕机 → 新 Pod 无法调度（但已有 Pod 不受影响）
  - API Server 宕机 → 集群控制平面不可用
  - 故障影响面：全集群

轻量级: 调度器是独立组件
  - 调度器宕机 → 可快速切换到备用实例
  - 已分配的沙箱继续运行
  - 故障影响面：仅新请求受影响，且恢复时间 <1s（无状态重启）
```

### 2.3 差异总结

| 维度 | K8s 传统调度 | 无状态轻量级调度 | 沙箱场景适用性 |
|------|-------------|----------------|-------------|
| 状态管理 | 强一致（etcd/Raft） | 最终一致（心跳同步） | **轻量级胜** — 沙箱无需强一致 |
| 调度复杂度 | O(N×K)，30+ 插件 | O(log N)，3 步操作 | **轻量级胜** — 约束少 |
| 资源模型 | 静态 Request/Limit | 动态槽位 | **轻量级胜** — 间歇性负载 |
| 通信路径 | 4+ 次网络往返 | 2 次网络往返 | **轻量级胜** — 延迟敏感 |
| 扩展性 | API Server 中心瓶颈 | 无状态可水平扩展 | **轻量级胜** — 高并发 |
| 全局优化 | 全集群最优调度 | 局部贪心+异步补偿 | **K8s 胜** — 复杂约束场景 |
| 生态集成 | 原生 K8s 生态 | 需额外适配层 | **K8s 胜** — 运维友好 |
| 故障恢复 | 控制平面自愈 | 进程级重启 | **K8s 胜** — 成熟度高 |

**关键结论**：沙箱场景的核心需求（低延迟、高并发、无状态执行）恰好是 K8s 传统调度的弱项，而无状态轻量级调度的优势所在。但 K8s 在全局优化和生态集成方面不可替代。因此，**两层调度是正确的架构选择**。

## 3. 两层调度机制设计

### 3.1 整体架构

```
                        沙箱创建请求
                             │
                             ▼
┌──────────────────────────────────────────────────────────────┐
│                   接入层 (Gateway / SDK)                       │
│                  统一入口，请求路由                              │
└──────────────────────────────────────────────────────────────┘
                             │
                    ┌────────┴────────┐
                    │   路由分流决策     │
                    ▼                 ▼
┌──────────────────────┐  ┌──────────────────────────────────┐
│  Layer 1             │  │  Layer 2                         │
│  轻量级快速调度        │  │  K8s 原生调度                     │
│  (Fast Path)         │  │  (Slow Path)                     │
│                      │  │                                  │
│  延迟: <1ms(调度)    │  │  延迟: 2-5s                      │
│  + 5-50ms(激活)      │  │                                  │
│  = ~50ms 端到端      │  │  触发条件:                        │
│                      │  │  - 预热池耗尽                     │
│  适用: 95%+ 的请求   │  │  - 需要特殊调度约束                │
│                      │  │  - 全局资源均衡                    │
└──────────────────────┘  └──────────────────────────────────┘
         │                          │
         ▼                          ▼
┌──────────────────────────────────────────────────────────────┐
│                   节点执行层                                   │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐        │
│  │ Node A  │  │ Node B  │  │ Node C  │  │ Node D  │        │
│  │ 预热池   │  │ 预热池   │  │ 预热池   │  │ 预热池   │        │
│  │ [■■□□]  │  │ [■□□□]  │  │ [■■■□]  │  │ [□□□□]  │        │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘        │
└──────────────────────────────────────────────────────────────┘
         │
         ▼
┌──────────────────────────────────────────────────────────────┐
│  协调控制器 (Reconciler)                                       │
│  周期: 5-10s                                                   │
│  职责: 预热池水位管理 / 资源全局优化 / Layer 1 ↔ Layer 2 同步    │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 Layer 1: 轻量级快速调度（Fast Path）

#### 3.2.1 核心数据结构

```python
@dataclass
class SlotPool:
    """节点槽位池 — 调度器在内存中维护的唯一状态"""
    pool_type: str              # "gpu-a100" | "cpu-only" | "arm-kunpeng" | ...
    total_slots: int            # 总槽位（预热池最大容量）
    free_slots: int             # 当前空闲槽位
    prewarmed_ready: int        # 已预热就绪的实例数
    nodes: PriorityQueue[NodeInfo]  # 按空闲槽位数排序的节点队列

@dataclass
class NodeInfo:
    """节点信息 — 通过心跳异步更新"""
    node_id: str
    pool_type: str
    free_slots: int
    prewarmed_ready: int
    gpu_utilization: float      # 0-1
    cpu_utilization: float
    last_heartbeat: float       # Unix timestamp
    endpoint: str               # 节点 Agent 通信地址
```

#### 3.2.2 调度算法

```
Algorithm: FastSchedule(request)
Input:  request = { pool_type, slots_needed, user_id }
Output: node_id | null

1. LOOKUP: pool ← slot_pools[request.pool_type]
   if pool == null → delegate to Layer 2

2. BEST_FIT: candidate ← pool.nodes.peek()
   if candidate.free_slots < request.slots_needed → delegate to Layer 2

3. ALLOCATE:
   pool.nodes.pop()
   candidate.free_slots -= request.slots_needed
   candidate.prewarmed_ready -= request.slots_needed
   pool.free_slots -= request.slots_needed
   pool.nodes.push(candidate)  // 重新排序

4. NOTIFY: async_send(candidate.endpoint, { action: "activate", slots: request.slots_needed })
   // 异步通知，不等响应，激活在节点本地完成

5. RETURN candidate.node_id

复杂度: O(log N)，其中 N 为同类型节点数
延迟: <1ms（纯内存操作 + 一次异步通知）
```

**关键设计决策**：

- **Best-Fit 而非 First-Fit**：选择空闲最多的节点（Best-Fit），而非第一个满足条件的节点。这避免了"碎片化"问题——多个节点各分配一点，导致每个节点都无法满足较大请求。
- **异步通知**：调度决策和节点激活解耦。调度器完成分配后立即返回，节点在本地异步激活沙箱。即使激活失败，也不影响调度决策的响应时间。
- **不区分"预热中"和"已预热"**：简化状态管理。如果请求到达时没有足够的已就绪实例，直接降级到 Layer 2，而不是等待预热完成。

#### 3.2.3 分流决策

```
Function: RouteRequest(request)
Returns: "fast_path" | "slow_path"

IF request.has_special_constraints():       // 亲和性/反亲和性/GPU型号指定
    RETURN "slow_path"

IF slot_pools[request.pool_type].free_slots >= request.slots_needed:
    RETURN "fast_path"

IF slot_pools[request.pool_type].prewarmed_ready >= request.slots_needed * 0.3:
    // 至少 30% 已就绪，其余可从温池快速恢复
    RETURN "fast_path"

RETURN "slow_path"
```

**决策依据**：宁可多发一些请求到 Layer 2（性能较差但正确），也不要在 Layer 1 做过于复杂的等待/重试逻辑。这保持了 Layer 1 的简单性，即其"快"的根本原因。

### 3.3 Layer 2: K8s 原生调度（Slow Path）

Layer 2 并非"被降级的备选方案"，而是承担着**全局资源优化**的关键职责。

#### 3.3.1 Layer 2 的职责

```
┌──────────────────────────────────────────────────────────────┐
│  Layer 2 的四大职责                                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 预热池补充 (Pool Replenishment)                           │
│     - 监控各节点预热池水位                                     │
│     - 通过 K8s 调度创建新的预热沙箱 Pod                        │
│     - 确保 Layer 1 始终有足够的槽位可用                         │
│                                                              │
│  2. 特殊约束调度 (Constrained Scheduling)                     │
│     - GPU 型号指定（A100 vs H100）                             │
│     - 节点亲和性/反亲和性                                      │
│     - 数据本地性（代码/模型预置在特定节点）                      │
│     - 多租户隔离策略                                          │
│                                                              │
│  3. 全局资源优化 (Global Optimization)                         │
│     - 跨节点负载均衡                                          │
│     - 弹性伸缩（新增/移除节点）                                │
│     - Spot 实例切换（成本优化）                                │
│                                                              │
│  4. 故障恢复 (Failure Recovery)                               │
│     - 节点故障时重新调度预热池                                  │
│     - 调度器故障时的状态恢复                                   │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

#### 3.3.2 预热池补充机制

这是两层调度协同最核心的机制：

```
协调控制器 (Reconciler)
    │
    │ 每 5-10s 执行一次
    ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 1: 收集各节点水位                                         │
│                                                              │
│  Node A: gpu-a100  水位 3/10  使用率 70%  → 需补充                                    │
│  Node B: gpu-a100  水位 7/10  使用率 30%  → 充足                                      │
│  Node C: cpu-only  水位 2/20  使用率 90%  → 需紧急补充                                 │
│  Node D: cpu-only  水位 18/20 使用率 10% → 可缩减                                     │
└──────────────────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 2: 计算补充策略                                          │
│                                                              │
│  维持水位 = max(基线水位, 历史峰值 * 1.2)                      │
│  补充数量 = 维持水位 - 当前水位                                 │
│  补充优先级 = 使用率高的节点优先                                │
│                                                              │
│  Node A: 维持=8, 当前=3, 补充=5, 优先级=高                     │
│  Node C: 维持=15, 当前=2, 补充=13, 优先级=紧急                  │
│  Node D: 维持=8, 当前=18, 缩减=10, 释放资源                    │
└──────────────────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 3: 通过 K8s API 执行                                     │
│                                                              │
│  创建 Sandbox CRD（预热模式）:                                 │
│  apiVersion: agent-sandbox/v1alpha1                           │
```

### 3.4 节点心跳与状态同步

```
节点 (每 1-3s)                    调度器
    │                               │
    │  Heartbeat {                  │
    │    node_id: "A",              │
    │    pool_type: "gpu-a100",     │
    │    free_slots: 5,             │──▶ 更新本地索引（无需响应）
    │    prewarmed: 5,              │
    │    gpu_util: 0.4,             │
    │    cpu_util: 0.6              │
    │  }                            │
    │                               │
    │◀──── ACK (可选，UDP)           │
    │                               │
    │  如果连续 3 次无 ACK:          │
    │  → 标记节点为不可用            │
    │  → 从槽位池中移除              │
    │  → 触发 Layer 2 补充          │
```

**设计要点**：
- 心跳是单向推送（节点 → 调度器），调度器不主动查询
- 使用 UDP 而非 TCP，避免连接管理开销
- 心跳丢失容忍：连续丢失 3 次（~9s）才标记不可用，避免网络抖动导致误判
- ACK 是可选的：调度器可以不回复，节点通过"连续 N 次心跳无调度分配"推断自己可能被隔离

### 3.5 延迟对比：端到端分析

| 阶段 | 纯 K8s 路径 | 两层调度（Fast Path 命中） | 两层调度（Slow Path） |
|------|-----------|------------------------|-------------------|
| 请求接入 | ~1ms | ~1ms | ~1ms |
| 路由决策 | N/A | <0.1ms | <0.1ms |
| 调度决策 | 100ms-1s | <1ms | 100ms-1s |

| **端到端** | **600ms-5s+** | **~50ms** | **600ms-5s** |

## 4. 关键技术决策

### 4.1 预热策略分层

```
┌─────────────────────────────────────────────────────────────────┐
│                     预热池三层设计                                 │
├────────────┬────────────────┬────────────────┬──────────────────┤
│            │   热池 (Hot)    │  温池 (Warm)    │  冷池 (Cold)      │
├────────────┼────────────────┼────────────────┼──────────────────┤
│ 状态       │ Running        │ Paused/CRIU    │ Image Ready Only  │
│ 激活延迟    │ 5-50ms         │ 50-300ms       │ 500ms-3s          │
│ 内存占用    │ 全量 (100%)    │ 压缩 (10-30%)  │ 仅镜像缓存 (~0%)  │
│ CPU 占用    │ 空转开销       │ 零             │ 零               │
│ 适用时机    │ 常规分配       │ 热池不足时补充   │ 突发峰值          │
│ 成本/收益比 │ 低成本/低延迟   │ 中成本/中延迟   │ 零成本/高延迟     │
├────────────┼────────────────┼────────────────┼──────────────────┤
│ 技术实现    │ 运行中的空沙箱   │ CRIU 快照/     │ 镜像预拉取到       │
│            │ 内存中等待分配   │ 内存冻结       │ 本地节点           │
├────────────┼────────────────┼────────────────┼──────────────────┤
│ 水位策略    │ 维持历史均值     │ 维持 30-50%     │ 维持 100%         │
│            │ × 1.2          │ 的热池水位      │ 节点覆盖          │
└────────────┴────────────────┴────────────────┴──────────────────┘

激活流程:
  热池:  [Running] ──注入用户代码──▶ [Active]     ~5-50ms
  温池:  [Paused]  ──内存恢复──▶ [Running]──▶ [Active]  ~50-300ms
  冷池:  [Image]   ──创建容器──▶ [Running]──▶ [Active]  ~500ms-3s
```

**水位管理策略**：

```python
class PoolManager:
    def calculate_watermark(self, pool_type: str) -> dict:
        """基于历史数据计算各层水位"""
        history = self.get_usage_history(pool_type, window="24h")

        baseline_hot = percentile(history.max_concurrent, 80)  # P80 峰值
        baseline_warm = baseline_hot * 0.5                       # 热池 50% 补偿
        baseline_cold = baseline_hot * 0.2                       # 突发缓冲

        return {
            "hot": ceil(baseline_hot * 1.2),    # 20% 冗余
            "warm": ceil(baseline_warm),
            "cold": ceil(baseline_cold),
        }
```

### 4.2 快速装箱算法选择

| 算法 | 描述 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **Best-Fit** | 选空闲最多的节点 | 减少碎片化，大请求更容易满足 | 可能导致"热点"节点 | **推荐默认** |
| First-Fit | 选第一个满足的节点 | O(1)，最快 | 碎片化严重 | 简单场景 |
| Worst-Fit | 选空闲最少的节点 | 均匀分布 | 碎片化，大请求容易失败 | 不推荐 |
| Spread | 尽量分散到不同节点 | 故障隔离好 | 网络开销大 | 对延迟不敏感 |

**推荐 Best-Fit**，原因：
1. 沙箱是同质化的（同类型资源池内可互换），不需要考虑亲和性
2. Best-Fit 自然聚合同类型请求到少数节点，预热效率更高
3. 大请求（如 GPU 密集推理）更容易满足

### 4.3 一致性与容错

```
问题: Layer 1 是最终一致的，可能出现什么问题？

场景 1: 两个调度器实例同时分配同一个槽位
  → 解决: 乐观并发控制（节点本地 CAS 操作）
  → 调度器分配 → 节点原子扣减 → 失败则调度器回滚索引

场景 2: 心跳延迟导致分配到已满的节点
  → 解决: 节点本地拒绝 + 调度器重试
  → 重试最多 1 次（换节点），仍失败则降级 Layer 2

场景 3: 调度器宕机
  → 解决: 多实例 + 无状态
  → 新实例从节点心跳快速重建索引（~3-10s）
  → 期间新请求全部走 Layer 2

一致性选择: 调度决策不追求强一致
  理由: 分配到一个"次优"节点的代价 < 每次调度的延迟开销
```

## 6. 决策记录

| 决策点 | 选择 | 原因 |
|--------|------|------|
| Layer 1 状态管理 | 内存索引 + 心跳同步 | 最终一致性已满足沙箱场景，避免 etcd 延迟 |
| 调度算法 | Best-Fit | 减少碎片化，同质化沙箱不需要亲和性 |
| 通信协议 | UDP 心跳 + gRPC 激活 | 心跳用 UDP 减少连接开销，激活用 gRPC 保证可靠性 |
| 预热策略 | 三层（热/温/冷） | 平衡延迟与资源成本 |
| Layer 1 实例数 | 3（高可用） | 无状态，多实例通过 gossip 同步索引 |
| 与 K8s 的关系 | 松耦合（仅通过 CRD 交互） | Layer 1 独立部署，不修改 K8s 核心代码 |
